# Задачи по закрытию архитектурного долга

Источник: разделы `Исторический компромисс / архитектурный долг` из [`architecture.md`](D:/GitHub/ui-kit/.agents/skills/arch-review/architecture.md).
Примечание: задачи, завязанные на хранение `dist` в git, удалены, так как `dist` находится в `.gitignore`.

## P0 — Критично для целостности поставки

- [ ] `ARCH-002` Закрепить обязательный тестовый gate для релиза.
      Критерий готовности: в release CI добавлен шаг запуска тестов (`vitest`/storybook tests), релиз не выполняется при падении тестов.

- [ ] `ARCH-003` Обновить проектную документацию релиза под фактический процесс.
      Критерий готовности: `AGENTS.md` и/или релизная документация не содержат устаревшего требования `NPM_TOKEN`, если используется OIDC/provenance; шаги релиза соответствуют workflow.

## P1 — Важно для стабильного API и предсказуемой интеграции

- [ ] `ARCH-004` Явно оформить стратегию поставки стилей.
      Критерий готовности: принято и зафиксировано одно из решений:

1. `index.js` гарантированно подтягивает стили;
2. стили публикуются отдельным официальным subpath export (например, `./style.css`) с инструкцией для потребителя.

- [ ] `ARCH-005` Согласовать `sideEffects` с реальным поведением style side-effects.
      Критерий готовности: значение `sideEffects` и структура импортов не приводят к случайному tree-shaking глобальных стилей в целевых bundler-ах; это подтверждено минимальным интеграционным smoke-тестом.

- [ ] `ARCH-006` Развести runtime и docs-артефакты в структуре исходников.
      Критерий готовности: docs-only файлы (`*.mdx`, при необходимости stories) вынесены в отдельный слой/каталог либо их роль как runtime-independent слоя явно описана в архитектурном документе.

- [ ] `ARCH-007` Явно определить статус `UiIcon` (public или internal).
      Критерий готовности: выбран и реализован один из вариантов:

1. `UiIcon` добавлен в public exports и документацию;
2. компонент переименован/переструктурирован как internal без двусмысленного public-сигнала.

- [ ] `ARCH-008` Формализовать границу internal API в Storybook.
      Критерий готовности: stories не зависят от внутренних модулей как от “псевдо-публичного API” без явной пометки; требования к использованию внутренних сущностей задокументированы. Типа stories должны зависеть только от публичного API?

## P2 — Улучшения консистентности design-system слоя

- [ ] `ARCH-009` Централизовать theme-логику между токенами и компонентными стилями.
      Критерий готовности: правила темизации (`light/dark`) описаны как единый слой (что задается токенами, что — компонентом), без противоречий между `tokens.scss` и локальными override.

- [ ] `ARCH-010` Минимизировать глобальные побочные эффекты из `tokens.scss`.
      Критерий готовности: влияние `body { background-color: var(--ui-bg); }` на хост-приложение либо устранено, либо вынесено в явно опциональный слой с документацией подключения.

- [ ] `ARCH-011` Снизить риск глобальных CSS-коллизий.
      Критерий готовности: выбран и задокументирован единый подход для изоляции стилей компонентов (строгий namespace/префиксация, scoped-стратегия или иная формальная политика).

- [ ] `ARCH-012` Развести документационные и тестовые Storybook-сценарии.
      Критерий готовности: тестовые сценарии явно маркируются/структурируются отдельно от docs stories, чтобы избежать смешения “витрины” и “проверки контракта”.

- [ ] `ARCH-013` Формализовать API-политику для базовых компонентов.
      Критерий готовности: зафиксировано, где библиотека использует props-only API, а где допускаются slots/emits; критерии выбора применяются последовательно к новым компонентам.

## Проверка закрытия долгов (Definition of Done)

- [ ] `ARCH-014` Добавить регулярную архитектурную валидацию в CI.
      Критерий готовности: есть автоматический или полуавтоматический чек-лист, который проверяет:
      наличие тестового gate, актуальность export map, согласованность документации релиза и API, а также соответствие генерируемых публичных типов исходникам. Это походу жесть и без LLM не обойтись? Или что-то реально можно автоматизировать скриптами?
